# 実データ化プラン

このプロトタイプ（ダミーデータの静的サイト）を、Wikipedia 由来の実データで動かすための計画。
グリルセッション（全13問）で合意した確定版。位置づけは **B（リアルなデモ）** — Wikipedia を一旦信頼し、正確性の考証は課さない。

## 方針の土台

- **水準は B（リアルなデモ）**。Wikipedia を信頼ソースとし、正確性検証は課さない。
- **更新は都度生成（オンデマンド）**。定期自動化はしない（新型投入が稀なため）。
- **自動収集の対象は「現行車種の諸元」だけ**。系譜ツリーは手書き（世代年・派生関係・現行/廃番は編集判断のため Wikipedia から機械抽出しない）。

## データ層（すべて JSON・`data/`）

| ファイル              | 誰が書く           | 中身                                                                       |
| --------------------- | ------------------ | -------------------------------------------------------------------------- |
| `data/genealogy.json` | 人（手書き）       | 系譜ツリー：ネームプレート・世代年リスト・派生関係・現行/廃番              |
| `data/lineup.json`    | 人（手書き）       | 比較表の現行車種マニフェスト。各エントリに `wikipediaTitle` と `targetGen` |
| `data/specs.json`     | 機械（生成）       | 諸元（数値ネイティブ）。人は触らない                                       |
| `data/raw/*.txt`      | 機械（キャッシュ） | 取得した Wikipedia 生ソース（再現・追跡用）                                |

### 諸元は徹底数値化

- 単位（mm / kg / PS / km/L）は**列定義側に1回だけ**持ち、セルは純粋な数値にする。
- 全長×全幅×全高 → **全長 / 全幅 / 全高の3列に分割**（number）。各寸法で最大/最小比較が効く。
- 価格帯 → `{ min, max }`（万円、number）。
- 発売年 → 系譜レイヤの世代年から引くので**諸元からは削除**。世代ラベルだけ残す。
- カテゴリ項目（ボディタイプ・エンジン・駆動方式）は文字列。

### 範囲・選定

- **少数キュレーション**：系譜6〜10系統、比較表6車種前後。網羅はしない。
- 比較表の車種は**系譜ツリーと独立**にキュレーション（ボディタイプの多様性を優先）。

## パイプライン（取得と抽出を分離）

1. **取得**：依存ゼロの https スクリプトで Wikipedia API から記事を取得し、`data/raw/` に保存（決定的・キャッシュ）。
2. **抽出**：**Claude Code が駆動**（Anthropic API スクリプト・SDK・API キー不要）。`data/raw/` の本文＋スキーマから `specs.json` を生成する。
3. この手順を **`.claude/skills/rebuild-specs/` スキル化**（`/rebuild-specs` で起動、リポジトリでバージョン管理）。

## フロントエンド

- **ビルド時に JSON を静的 import**（実行時 fetch なし・CORS なし・ローディング状態なし）。
- `buildLineageView()` を **`buildLineageView(genealogy, lineup, specs)` の純粋関数**に作り替える。
  - データは import した JSON、関数は座標・コネクタ・grid 幅・最大/最小ハイライトの計算だけを担う。
  - 既存のジオメトリ計算とテストは維持（引数渡しに変更）。
- Tailwind 化とは直交（静的スタイル=`className`、動的スタイルのみ `css()`）。

## 検証

- **vitest のデータ整合テスト**を追加（依存ゼロ・CI に乗る）。
  - lineup ↔ specs の対応、数値の有限性、必須フィールド、派生の親の存在などを検査。
  - 壊れたデータは静的 import でブラウザに持ち込む前に、プッシュ前の `npm run check` / CI で落とす。

## 実装時に詰める細部

- `genealogy.json` / `specs.json` の正確なフィールド名とスキーマ形状（interface の再設計）。
- 諸元の列定義（単位・数値/文字列の別・比較対象フラグ）をどこに置くか（`lineup.json` かスキーマ定数か）。
- Wikipedia API の具体的なエンドポイント（`action=query&prop=extracts` 等）。

## 実装順序（オーケストレーション）

1. **スキーマ／契約の確定**（linchpin・オーケストレータが担当）：`data/` の TS 型と JSON 形状、列定義の置き場所を定義。
2. 契約確定後に並行実装（サブエージェント）：
   - フロント：`buildLineageView` の作り替え＋コンポーネント接続。
   - パイプライン：取得スクリプト＋`rebuild-specs` スキル＋`specs.json` 生成。
   - 検証：vitest データ整合テスト。
   - 手書きデータ：`genealogy.json` / `lineup.json` の作成。
